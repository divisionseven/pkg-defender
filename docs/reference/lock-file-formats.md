# Lock File Formats

> **Purpose:** Reference for every lock file format PKG-Defender can parse for auditing ‚Äî covering format details, transitive dependency coverage, and auditability characteristics.

> For the full list of package managers and registry adapters, see the [Data Dictionary](../reference/data-dictionary.md).

---

## Currently Parsed Formats

The table below lists all lock file formats PKG-Defender can currently parse for auditing. Each format has a dedicated parser function in `src/pkg_defender/core/parsers.py`. For lock files that are not yet parsed (e.g. `Cargo.lock`, `Gemfile.lock`), see the [Other Lock Files](#other-lock-files) section below.

| Lock File           | Ecosystem         | Package Manager(s)               | Format     | Transitive Deps | Notes                                                                                                                                                                                         |
| ------------------- | ----------------- | -------------------------------- | ---------- | --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `requirements.txt`  | Python / PyPI     | `pip`, `pip3` (pip-tools)        | Plain text | Partial         | Only exact pins (`pkg==X.Y.Z`) are extracted via `parse_requirements_txt()`. Not a true lockfile unless generated by `pip-tools` with `--generate-hashes`. No hash verification is performed. |
| `uv.lock`           | Python / PyPI     | `uv`                             | TOML       | Full            | Rich lockfile with full dependency graph, hashes, and markers. Highly auditable. Parsed by `parse_uv_lock()`.                                                                                 |
| `poetry.lock`       | Python / PyPI     | `poetry`                         | TOML       | Full            | Complete dep tree with hashes. Parsed by `parse_poetry_lock()`.                                                                                                                               |
| `Pipfile.lock`      | Python / PyPI     | `pipenv`                         | JSON       | Full            | Contains hashes for all direct and transitive deps. Parsed by `parse_pipfile_lock()`.                                                                                                         |
| `package-lock.json` | JavaScript / Node | `npm`                            | JSON       | Full            | v2/v3 formats supported by `parse_package_lock()`. Does **not** cover `npm-shrinkwrap.json`.                                                                                                  |
| `yarn.lock`         | JavaScript / Node | `yarn` (v1 Classic, v2‚Äìv4 Berry) | Custom     | Full            | Parsed by `parse_yarn_lock()` using two regex patterns (`_YARN_HEADER_RE` and `_YARN_VERSION_RE`) that work as a state-machine pair. No version-specific dispatch; the same header regex handles both v1 and Berry formats.                                                   |
| `pnpm-lock.yaml`    | JavaScript / Node | `pnpm`                           | YAML       | Full            | Stores flattened and nested dep graph. Parsed by `parse_pnpm_lock()`.                                                                                                                         |

| `Cartfile.resolved`      | Swift / Obj-C           | `carthage`            | No parser yet. Custom text format, direct deps only.                                                                                      |
| `Brewfile.lock.json`     | System / Mixed          | `brew` (Homebrew)     | No parser yet. JSON format.                                                                                                               |
| `flake.lock`             | System / Mixed          | `nix`                 | No parser yet. JSON format.                                                                                                               |

### Formats Without Lockfile Support

The following ecosystems do not provide a standard lockfile for auditing:

| Ecosystem              | Package Manager(s) | Notes                                                                                                                   |
| ---------------------- | ------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| Java / JVM             | `maven` (mvn)      | No native lockfile. `pom.xml` declares deps; resolution is build-time only. Some teams use the `maven-lockfile` plugin. |
| System (Debian/Ubuntu) | `apt`, `apt-get`   | No lockfile. Relies on OS package signing infrastructure.                                                               |
| System (RHEL/Fedora)   | `dnf`, `yum`       | No lockfile. Relies on OS package signing infrastructure.                                                               |
| System (Arch)          | `pacman`           | No lockfile. Relies on OS package signing infrastructure.                                                               |

## Format Detection Notes

### Binary Formats

- **`bun.lockb`** is a binary format used by Bun v1. It requires running `bun bun` or using Bun's built-in lockfile parser to convert it to a readable form. The newer `bun.lock` text format (Bun v1.1.x+) should be preferred when available. Neither format is currently parsed by PKG-Defender.

### `yarn.lock` ‚Äî v1 vs. Berry (v2‚Äìv4)

- Both v1 (Classic) and Berry formats are handled by the regex-based parser `parse_yarn_lock()`.
- **No version detection** is performed ‚Äî the same regex pattern matches dependency headers in both formats.
- The parser does not distinguish between v1 and Berry-specific syntax features.

### `requirements.txt` ‚Äî Partial Lockfile

- Not a true lockfile unless generated by `pip-tools` with pinned versions and `--generate-hashes`.
- PKG-Defender extracts only exact version pins (`package==X.Y.Z`) ‚Äî range specifiers and options are skipped.
- No hash verification is performed; the file provides version pins but no integrity verification.

### Maven ‚Äî No Standard Lockfile

- Maven has no native lockfile mechanism. Dependency resolution happens at build time from `pom.xml`.
- Auditing Maven projects requires parsing `pom.xml` and resolving the dependency tree ‚Äî which requires a Maven invocation or a custom resolver.
- The `maven-lockfile` community plugin provides an opt-in lockfile, but it is not standard.

### Gradle ‚Äî Opt-In Lockfiles

- `gradle.lockfile` must be explicitly enabled via Gradle's `dependencyLocking` feature.
- If `gradle.lockfile` is absent, auditing requires parsing `build.gradle` / `build.gradle.kts`.

### `Cargo.lock` ‚Äî Binary vs. Library Convention

- `Cargo.lock` is always committed by convention for **binary crates** (applications).
- **Library crates** commonly `.gitignore` the lockfile, making it unavailable for auditing.
- `Cargo.lock` is **not currently parsed** ‚Äî the Cargo adapter only handles registry lookups (publish times, version lists).

### Parser Architecture

Lock file parsing is concentrated in a single module: `src/pkg_defender/core/parsers.py`.
All currently supported formats are dispatched through `parse_lock_file()` which routes
by filename. Detection of which lock file is present is handled by `detect_lock_file()`,
which checks for the existence of known filenames in priority order (defined in
`LOCK_FILE_NAMES`) ‚Äî no regex or glob matching is used. Package manager adapters
(Bun, Cargo, etc.) do **not** implement lock file parsing ‚Äî they handle command
parsing, registry lookups, and exec argument construction.
